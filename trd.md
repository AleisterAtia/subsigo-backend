# Technical Requirements Document (TRD)

**Product:** SI-TEPAT — Sistem Integrasi e-KTP Tepat Sasaran
**Component:** Backend API Service (`subsigo-backend`)
**Companion document:** [`prd.md`](./prd.md) (Product Requirements, v1.1)
**TRD Version:** 1.0
**Status:** Reflects the system as currently deployed on Vercel + Neon.

---

## 1. Purpose & Scope

This document describes the **technical design of the SI-TEPAT backend** as it is
actually implemented today, and defines the **remaining gaps** required to take the
system from a working MVP to a complete, production-ready service.

**In scope:** the Golang backend API, its data model, authentication, the
concurrency-safe claim flow, configuration, and deployment on Vercel.

**Out of scope (separate repositories):** the Flutter mobile app (field officer)
and the Next.js admin dashboard. This TRD only defines the **contracts** the backend
exposes to those clients.

## 2. System Architecture

### 2.1 High-level

```
 ┌──────────────────────┐        ┌──────────────────────┐
 │  Mobile App (Flutter)│        │ Admin Dashboard      │
 │  Field Officer + NFC │        │ (Next.js, browser)   │
 └──────────┬───────────┘        └──────────┬───────────┘
            │  HTTPS / JSON + JWT            │  HTTPS / JSON + JWT (CORS)
            └───────────────┬───────────────┘
                            ▼
              ┌───────────────────────────────┐
              │  Vercel (Go Framework Preset) │
              │  Long-running Fiber web server│
              │  cmd/api/main.go  ← entrypoint│
              └───────────────┬───────────────┘
                              │  database/sql (pgx) via GORM
                              ▼
              ┌───────────────────────────────┐
              │  Neon Serverless PostgreSQL   │
              │  (pooled connection / PgBouncer)│
              └───────────────────────────────┘
```

### 2.2 Deployment model (important clarification vs PRD)

The PRD §3.B asks for a "Serverless Function" on Vercel. In practice, Vercel's
**Go Framework Preset** detects `go.mod` + `cmd/api/main.go` and runs the app as a
**long-running web server on Vercel Fluid Compute**, injecting a `PORT` env variable
the server must bind to. This still satisfies the serverless *objectives* —
auto-scaling, scale-to-zero, zero server management — but it is a Vercel-managed
**web service**, not a discrete `@vercel/go` function. The PRD wording should be
updated to "Vercel-managed Go web service (Fluid Compute)".

### 2.3 Layered structure

```
cmd/api/        HTTP entrypoint (Vercel runs this); reads PORT, calls server.New()
cmd/migrate/    Standalone schema migration runner (GORM AutoMigrate)
cmd/seed/       Standalone seed of initial admin/officer users
internal/server         Builds the Fiber app: middleware, routes, DI
internal/config         Env config + resolvePort() (PORT → APP_PORT → 8080)
internal/handlers       HTTP layer (request parse/validate, status codes)
internal/services       Business logic (auth, admin, claim)
internal/repositories   Data access (GORM)
internal/middlewares    RequireAuth (JWT) + RequireRole (RBAC)
internal/models         Entities, constants, AutoMigrate registration
pkg/database            Neon connection + pool tuning
pkg/token               JWT generate/parse (HS256)
pkg/hash                bcrypt hash/check
```

**Flow:** `handler → service → repository → model`, wired by dependency injection in
`internal/server.New()`.

## 4. Data Model

All tables use `uuid` primary keys generated by Postgres `gen_random_uuid()`.
Timestamps are stored in **UTC** (`NowFunc` overridden in `pkg/database`).

### 4.1 `users` — system accounts (officers & admins)

| Column | Type | Notes |
|---|---|---|
| id | uuid PK | |
| username | varchar(64) | unique, not null |
| password_hash | varchar(255) | bcrypt, never serialized (`json:"-"`) |
| role | varchar(16) | not null, default `merchant`; values: `admin`, `merchant` |
| merchant_name | varchar(128) | only relevant for `merchant` (SPBU/agent location) |
| is_active | bool | not null, default true — soft-disable; inactive accounts cannot log in |
| created_at / updated_at | timestamp | |

### 4.2 `citizens` — population data (simulated; no Dukcapil)

| Column | Type | Notes |
|---|---|---|
| id | uuid PK | |
| nik | varchar(16) | unique, not null (16-digit National ID) |
| nfc_uid | varchar(64) | unique, not null — **primary lookup key** on e-KTP tap |
| name | varchar(128) | not null |
| is_eligible | bool | not null, default true (admin toggles subsidy eligibility) |
| created_at / updated_at | timestamp | |

### 4.3 `subsidy_quotas` — per-citizen quota per commodity per period

| Column | Type | Notes |
|---|---|---|
| id | uuid PK | |
| citizen_id | uuid | not null, part of unique index `idx_quota_unique` |
| commodity | varchar(32) | not null, part of unique index; values: `LPG_3KG`, `PERTALITE` |
| period | varchar(7) | not null, part of unique index; format `YYYY-MM` |
| quota_total | int | not null, default 0 |
| quota_remaining | int | not null, default 0 — **the row locked during a claim** |
| created_at / updated_at | timestamp | |

**Unique constraint:** `(citizen_id, commodity, period)` — lets quotas reset per
month without overwriting prior periods' history.

### 4.4 `transactions` — audit log of every claim attempt

| Column | Type | Notes |
|---|---|---|
| id | uuid PK | |
| citizen_id | uuid NULL | nullable: unregistered UID taps are still logged |
| nfc_uid | varchar(64) | not null, indexed |
| user_id | uuid | not null, indexed (the serving officer) |
| commodity | varchar(32) | not null |
| status | varchar(16) | not null; values: `success`, `rejected` |
| reason | varchar(128) | rejection reason (empty on success) |
| merchant_name | varchar(128) | |
| created_at | timestamp | indexed (for `ORDER BY created_at DESC` monitoring) |

Schema is applied by `cmd/migrate` (GORM `AutoMigrate`), kept separate from the API
runtime so multiple serverless instances never race to migrate.

## 5. API Specification

Base URL (production): `https://subsigo-backend.vercel.app`
All bodies are JSON. Protected routes require `Authorization: Bearer <jwt>`.

| Method | Path | Auth | Purpose |
|---|---|---|---|
| GET | `/health` | public | Liveness + DB ping |
| POST | `/api/v1/auth/login` | public | Authenticate → JWT (rejects inactive accounts with 403) |
| GET | `/api/v1/me` | any role | Current token's identity |
| GET | `/api/v1/admin/citizens?search=&page=&limit=` | admin | Paginated list + search (NIK/UID/name) |
| POST | `/api/v1/admin/citizens` | admin | Register citizen (UID ↔ NIK) |
| GET | `/api/v1/admin/citizens/:id` | admin | Citizen detail incl. quotas |
| PATCH | `/api/v1/admin/citizens/:id/eligibility` | admin | Toggle eligibility |
| POST | `/api/v1/admin/citizens/:id/quotas` | admin | Set/reset quota for a period |
| GET | `/api/v1/admin/users?search=&page=&limit=` | admin | Paginated list of accounts |
| POST | `/api/v1/admin/users` | admin | Create officer/admin account |
| PATCH | `/api/v1/admin/users/:id` | admin | Update role/merchant/password/is_active |
| GET | `/api/v1/admin/transactions` | admin | Filtered + paginated claim history |
| POST | `/api/v1/claims` | merchant | Process a subsidy claim |

**Pagination response shape** (list endpoints — citizens, users, transactions):
```json
{ "data": [ ... ], "pagination": { "page": 1, "limit": 20, "total": 137 } }
```
Defaults: `page=1`, `limit=20` (max `100`).

**Transaction filters** (`GET /api/v1/admin/transactions`): `status` (`success`|`rejected`),
`commodity` (`LPG_3KG`|`PERTALITE`), `user_id` (uuid), `merchant_name` (ILIKE),
`from`/`to` (`YYYY-MM-DD`, interpreted in **WIB**; `to` is inclusive of the whole day),
plus `page`/`limit`.

### 5.1 Representative payloads

**POST `/api/v1/auth/login`**
```json
// req
{ "username": "admin", "password": "admin123" }
// res 200
{ "token": "<jwt>", "user": { "id": "...", "username": "admin", "role": "admin", "merchant_name": "" } }
```

**POST `/api/v1/admin/citizens`** → `201`
```json
{ "nik": "3201234567890001", "nfc_uid": "04A1B2C3D4E5F6", "name": "Budi Santoso" }
```

**POST `/api/v1/admin/citizens/:id/quotas`** → `201`
```json
{ "commodity": "LPG_3KG", "period": "2026-06", "quota_total": 5 }  // period optional → current WIB month
```

**POST `/api/v1/admin/users`** → `201`
```json
// req (password ≥ 6, ≤ 72 chars; role: admin|merchant)
{ "username": "petugas2", "password": "rahasia123", "role": "merchant", "merchant_name": "SPBU 34-402" }
// res — password_hash is never serialized
{ "id": "...", "username": "petugas2", "role": "merchant", "merchant_name": "SPBU 34-402", "is_active": true }
```

**PATCH `/api/v1/admin/users/:id`** → `200` (all fields optional)
```json
{ "is_active": false }            // soft-disable an officer
{ "password": "passwordbaru" }    // reset password
{ "role": "admin", "merchant_name": "" }
```

**POST `/api/v1/claims`** → `200`
```json
// req
{ "nfc_uid": "04A1B2C3D4E5F6", "commodity": "LPG_3KG" }
// res (success)
{ "status": "success", "message": "Klaim berhasil", "quota_remaining": 4, "transaction_id": "..." }
// res (business rejection — still HTTP 200)
{ "status": "rejected", "message": "Kuota subsidi sudah habis", "quota_remaining": 0, "transaction_id": "..." }
```

### 5.2 Error/status conventions

- **Business rejections** of a claim (unregistered UID, not eligible, quota empty,
  quota not configured) return **HTTP 200** with `status: "rejected"` so the mobile
  app only needs to read one field.
- **Technical/validation errors** use 4xx/5xx with `{ "error": "<message>" }`
  (`400` bad input, `401` auth, `403` wrong role, `404` not found, `409` duplicate,
  `500` internal, `503` startup/DB unavailable).

## 6. Authentication & Authorization

- **Login:** username/password → bcrypt verify → JWT signed HS256.
- **JWT claims:** `uid` (user id), `role`, `mn` (merchant name), `iat`, `exp`
  (`JWT_EXPIRE_HOURS`, default 24h).
- **`RequireAuth`** middleware: parses `Bearer` token, rejects non-HMAC algorithms
  (mitigates `alg:none`), stores `userID`/`role`/`merchantName` in request context.
- **`RequireRole`** middleware: route-level RBAC. `/admin/*` requires `admin`;
  `/claims` requires `merchant`.

## 7. Core Logic — Concurrency-Safe Claim (PRD §3.B "Quota Management")

`ClaimService.Claim` runs the critical path inside **one DB transaction** with
**row-level locking** to prevent double-claims:

1. Validate commodity (`LPG_3KG` / `PERTALITE`).
2. Compute `period = now().UTC() → "YYYY-MM"`.
3. **BEGIN TX**
   a. Look up citizen by `nfc_uid`. Not found → record `rejected` ("KTP tidak terdaftar").
   b. Not eligible → record `rejected` ("Warga tidak layak…").
   c. `SELECT … FOR UPDATE` the matching `subsidy_quotas` row. Concurrent claims for
      the **same** quota row **block here** until the first commits, then read fresh.
      Not found → record `rejected` ("Kuota belum diatur").
   d. `quota_remaining <= 0` → record `rejected` ("Kuota sudah habis").
   e. Decrement `quota_remaining`, save, record `success`.
4. **COMMIT** — rejections are committed too (full audit trail).

This guarantees that even simultaneous taps of the same e-KTP on two devices cannot
over-draw the quota.

## 9. Deployment & Operations

- **Build/run:** Vercel Go Framework Preset → runs `cmd/api/main.go`, server binds to

## 11. Gap Analysis — What's Needed for a "Perfect" System

The MVP meets every backend-relevant PRD requirement. The gaps below are what remain
to make it **complete and production-grade**. Each gap lists priority, the risk it
addresses, a proposed solution, and acceptance criteria.

**Priority legend:** **P0** = required for full functional completeness (the admin
dashboard / officer flows cannot be fully delivered without it); **P1** = production
reliability & security hardening; **P2** = polish / future enhancement.

### P0 — Functional completeness

> **Status: P0 CLOSED (Phase 1 implemented).** GAP-01 … GAP-04 below are done; details
> retained for traceability. The dashboard/mobile contracts they unlock are in §5.

#### GAP-01 · Citizen read/search/list endpoints (P0) — ✅ DONE
- **Was:** only `POST` (create) and `PATCH` (eligibility) existed; no way to list,
  search, or view a single citizen and their quotas.
- **Implemented:**
  - `GET /api/v1/admin/citizens?search=&page=&limit=` — paginated list + ILIKE search
    on NIK/UID/name (`CitizenRepository.List`).
  - `GET /api/v1/admin/citizens/:id` — detail with quotas preloaded
    (`CitizenRepository.FindByIDWithQuotas`).
- **Acceptance met:** dashboard can search a citizen by NIK/UID and show their quotas.

#### GAP-02 · User/officer management endpoints (P0) — ✅ DONE
- **Was:** users only existed via `cmd/seed`; no API to create/list/disable accounts.
- **Implemented:** `POST/GET/PATCH /api/v1/admin/users` (admin-only) via new
  `UserService`/`UserHandler`. bcrypt-hashed passwords (length 6–72), role validation,
  and an `is_active` flag for soft-disable. Login now rejects inactive accounts (403).
- **Acceptance met:** an admin can create a new officer from the dashboard and that
  officer can log in; disabling sets `is_active=false` and blocks future logins.
- **Note:** mid-session JWT revocation is still GAP-08 (stateless token stays valid
  until expiry) — disabling blocks the *next* login, not an already-issued token.

#### GAP-03 · Transaction monitoring filters & pagination (P0) — ✅ DONE
- **Was:** `GET /admin/transactions?limit=N` only (capped at 200, no offset).
- **Implemented:** `TransactionFilter` supporting `status`, `commodity`, `user_id`,
  `merchant_name`, `from`/`to` (date range), plus `page`/`limit` and a total count
  (`{data, pagination:{page,limit,total}}`).
- **Acceptance met:** dashboard can filter a day's rejected LPG claims for one SPBU
  and page through results.

#### GAP-04 · Quota period strategy alignment (P0) — ✅ DONE
- **Was:** a claim used the **current UTC month** as `period`; UTC vs WIB month
  boundaries could resolve a near-midnight claim to the wrong month → silent rejects.
- **Implemented:** period derivation centralized in `models.CurrentPeriod()` using a
  fixed **WIB (UTC+7)** zone, used by BOTH the claim flow and the admin quota default,
  so a quota set for the active WIB month is always honored. `SetQuota` also validates
  the `YYYY-MM` format. **Strategy:** claims always use the current WIB month; admins
  set/reset quota for that (or a future) month.
- **Acceptance met:** a quota set for the active period is honored; timezone is
  deterministic (FixedZone, no tzdata dependency) and documented.

### P1 — Reliability & security hardening

#### GAP-05 · Automated tests (P1)
- **Current:** no `*_test.go` files.
- **Risk:** the concurrency-critical claim logic and RBAC are unverified; regressions
  ship silently.
- **Proposed:** unit tests for `ClaimService` (each rejection branch + success),
  table-driven validation tests, and an **integration test that fires concurrent
  claims** against a real/throwaway Postgres to prove the `FOR UPDATE` guard.
- **Acceptance:** `go test ./...` covers claim branches; a concurrency test asserts
  quota never goes negative under N parallel claims.

#### GAP-06 · Rate limiting & brute-force protection on login (P1)
- **Current:** `/auth/login` is unthrottled.
- **Risk:** credential brute-force.
- **Proposed:** per-IP + per-username rate limiting (Fiber `limiter` middleware or a
  store-backed limiter), plus exponential backoff/lockout after repeated failures.
- **Acceptance:** repeated failed logins are throttled and logged.

#### GAP-07 · Production CORS & secrets posture (P1)
- **Current:** `CORS_ALLOW_ORIGINS` defaults to `*`; seed passwords are weak
  (`admin123`, `petugas123`).
- **Risk:** overly permissive CORS in prod; default creds if not rotated.
- **Proposed:** require explicit origin(s) in prod; force a password change / generate
  strong seed secrets; document a "first admin bootstrap" procedure.
- **Acceptance:** prod env restricts CORS to the dashboard origin; no default password
  remains usable.

#### GAP-08 · Token lifecycle: refresh & revocation (P1)
- **Current:** stateless JWT, 24h, no refresh, no revocation. A role/eligibility
  change or a disabled officer stays valid until expiry.
- **Risk:** can't immediately revoke access (e.g., dismissed officer).
- **Proposed:** short-lived access token + refresh token, or a token version / denylist
  checked in `RequireAuth`; honor `is_active` on each request.
- **Acceptance:** disabling a user invalidates their access within the access-token TTL.

#### GAP-09 · Cold-start latency mitigation (P1)
- **Current:** Neon free-tier scales to zero; first request after idle is slow,
  tensioning with the "millisecond" target.
- **Proposed:** use a Neon plan without scale-to-zero for production, and/or a
  scheduled warm-up ping; document expected cold-start behavior.
- **Acceptance:** p95 latency under load excludes cold start; cold-start path documented.

#### GAP-10 · CI pipeline + observability (P1)
- **Current:** no CI gate (lint/test/build) before deploy; logging is basic, no metrics
  or error tracking.
- **Proposed:** GitHub Actions running `go vet`, `golangci-lint`, `go test`, `go build`
  on PRs; add structured logging + an error monitor (e.g. Sentry) and basic metrics.
- **Acceptance:** PRs blocked on failing checks; runtime errors are captured centrally.

| Gap | Summary |
|---|---|

## 13. Acceptance for "Production-Ready"

gaps GAP-05/06/07/08 are closed, CI is enforced (GAP-10), and the PRD §3.B wording is
